## 1. 文档定位

本文面向项目维护者和二次开发者，基于当前仓库状态整理，重点说明：

- 代码实际位于哪里。
- 开发环境该如何准备。
- 后端、前端、打包链路分别如何运行。
- 配置、任务参数、接口契约当前是怎样实现的。

注意：仓库根目录主要保存文档、统一 Python 环境和补充资源，真正的运行项目目录是 `Game_screenshotReed_Autowork/`。

---

## 2. 项目概览

`StellasoraMaster` 是一个面向《星塔旅人》的自动化辅助项目，整体采用：

- Python 自动化核心
- Flask 后端接口与页面服务
- Vue 3 + Vite 前端控制台
- ADB + 图像识别 + OCR 驱动任务执行

核心能力包括：

- 自动执行日常任务，可勾选子任务。
- 自动爬塔，支持属性、模式、次数、是否保留记录等参数。
- 实时画面监控（MJPEG）。
- ADB 配置、模拟器配置组、截图方式配置。
- 日志轮询与导出。
- 版本检查与在线更新。

## 3. 仓库结构与职责

### 3.1 仓库根目录

- `README.md`：项目入口说明。
- `开发文档.md`：本文档。
- `用户使用指南.md`：面向使用者的操作说明。
- `requirements.txt`：根目录推荐开发依赖，版本更完整，包含打包相关依赖。
- `.venv/`：当前仓库已经存在的统一 Python 虚拟环境目录。
- `Game_screenshotReed_Autowork/`：真正的运行项目目录。

### 3.2 运行项目目录 `Game_screenshotReed_Autowork/`

- `launcher.py`
  - 统一启动入口。
  - 默认启动主程序。
  - 传入 `--update`、`update` 或 `update.py` 时切换到更新流程。
- `run_app.py`
  - 非打包环境启动入口。
  - 会自动检查最小缺失依赖并通过清华镜像安装。
  - 自动寻找可用端口，从 `5000` 开始顺延。
  - 服务启动后自动打开浏览器。
- `update.py`
  - 在线更新逻辑。
  - 负责读取远端 `version.json`、下载补丁包、覆盖安装目录、更新版本号并重启主程序。
- `config.json`
  - 运行期配置文件。
  - 由设置页面和任务启动流程共同读写。
- `version.txt`
  - 本地版本号。
- `package.ps1`
  - 打包脚本。
- `StellasoraMaster.spec`
  - PyInstaller 规范文件。

### 3.3 自动化核心 `Game_screenshotReed_Autowork/core/`

- `base_task.py`
  - 任务基类。
  - 初始化截图、点击、滑动工具。
  - 根据 `server_lang` 选择 `templates_zh-CN/` 或 `templates_zh-Hant/`。
  - 提供 `click_until_appear`、`wait_until_appear`、可中断睡眠等复用能力。
- `config.py`
  - 配置加载、缓存、持久化和路径标准化。
- `mumu_screenshot.py`
  - 单例截图流。
  - 支持 `PNG`、`RAW`、`SCRCPY` 三种模式。
  - 内置连接检查、超时恢复、降级与重启逻辑。
- `mumu_click.py`
  - ADB 点击封装。
- `slide.py`
  - 滑动封装。
- `ocr_tool.py`
  - OCR 封装。
- `start_game.py`
  - 启动游戏并进入主界面。
- `daily_tasks.py`
  - 日常任务主流程。
- `tower_climber.py`
  - 自动爬塔主流程。
- `show_detector.py`、`start_icon_detector.py`
  - 模板识别相关能力。

### 3.4 后端与前端

- `Game_screenshotReed_Autowork/webapp/app.py`
  - Flask 主服务。
  - 提供任务、监控、日志、配置、更新相关接口。
  - 内部维护全局任务状态和截图工具实例。
- `Game_screenshotReed_Autowork/webapp/frontend/`
  - Vue 3 + Vite 前端源码目录。
- `Game_screenshotReed_Autowork/webapp/static/`
  - 前端构建产物目录。
  - Flask 根路径 `/` 默认直接返回这里的 `index.html`。

### 3.5 模板与调试目录

- `templates_zh-CN/`、`templates_zh-Hant/`
  - 两套语言模板资源目录。
  - 新增图标模板时通常需要两套同时维护。
- `test/`、`tower_climber_test/`
  - 当前仓库中为空目录。
  - 代码里更多把它们当作调试输出目录，而不是自动化测试目录。

## 4. 当前运行架构

### 4.1 启动链路

常见启动顺序如下：

1. 执行 `launcher.py`。
2. 若命中更新参数，则进入 `update.py`。
3. 否则调用 `run_app.main()`。
4. `run_app.py` 会在非打包环境下自动补安装最小依赖。
5. 导入 `webapp.app` 中的 Flask 应用。
6. 自动寻找空闲端口并启动服务。
7. 浏览器自动打开 `http://127.0.0.1:<port>/`。

### 4.2 任务执行链路

前端点击“开始执行”后：

1. 调用 `POST /task/start`。
2. 后端根据 `type` 和附加参数构建任务。
3. 启动前会调用截图工具的 `preflight_for_task()` 做 ADB 连接检查。
4. 创建后台线程执行任务。
5. 任务状态在 `idle / running / paused / stopped / finished` 间切换。
6. 前端通过 `/task/status` 轮询状态，并用 `/logs` 轮询日志。

### 4.3 预览与截图流链路

项目只有一个全局截图流实例，由 `MumuScreenshot` 单例维护。

- 预览页通过 `/stream/start` 获取长期消费者占用。
- `/video_feed` 返回 MJPEG 图像流。
- 日常任务和爬塔任务在运行时也会 `acquire_stream()`，避免重复起停底层截图流。
- `RAW` 失败时后端会自动回退到 `PNG`。
- `SCRCPY` 连续不稳定时会自动降级到 `RAW`。

当前前端设置页只允许选择：

- `PNG`
- `RAW`

后端仍保留 `SCRCPY` 支持，但前端会把旧的 `SCRCPY` 配置自动归一化为 `RAW`。

### 4.4 日志链路

`webapp/app.py` 会同时接管：

- `print`
- `logging`

日志会进入内存队列，前端通过 `/logs` 增量拉取，并可通过 `/logs/export` 导出文本。

## 5. 开发环境准备

### 5.1 推荐环境

当前项目明显偏向 Windows 开发和打包环境，原因包括：

- ADB 路径和示例均为 Windows 路径。
- `package.ps1` 为 PowerShell 打包脚本。
- PyInstaller 打包目标为 Windows 可执行文件。
- 更新流程默认查找 `Update.exe`、`StellasoraMaster.exe`。

### 5.2 Python 依赖

当前仓库有两份依赖描述：

- 根目录 `requirements.txt`
  - 推荐用于开发环境初始化。
  - 版本更完整，包含 `PyInstaller`、`PyQt5` 等额外依赖。
- `Game_screenshotReed_Autowork/requirements.txt`
  - 更像最小运行依赖列表。
  - 适合快速了解运行时必须组件，但不等价于完整开发环境。

推荐从仓库根目录初始化环境：

```powershell
python -m venv .venv
.\.venv\Scripts\pip install -r .\requirements.txt
```

### 5.3 前端依赖

前端源码位于 `Game_screenshotReed_Autowork/webapp/frontend/`：

```powershell
Set-Location .\Game_screenshotReed_Autowork\webapp\frontend
npm ci
```

当前 `package.json` 主要依赖为：

- `vue`
- `vite`
- `@vitejs/plugin-vue`

## 6. 常用开发启动方式

### 6.1 一体化运行（最接近日常使用）

适合验证“打包前的完整运行形态”，即 Flask 直接服务构建后的前端静态资源。

```powershell
Set-Location .\Game_screenshotReed_Autowork
..\.venv\Scripts\python .\launcher.py
```

如果已经激活虚拟环境，也可以直接：

```powershell
python .\launcher.py
```

特点：

- 会自动打开浏览器。
- 默认从 `5000` 起找可用端口。
- 前端页面来自 `webapp/static/`。
- 缺失最小依赖时，`run_app.py` 会尝试自动安装。

### 6.2 后端固定端口运行

适合前端热更新联调时使用。

```powershell
Set-Location .\Game_screenshotReed_Autowork
..\.venv\Scripts\python .\webapp\app.py
```

默认监听：

`http://127.0.0.1:5000/`

注意：

- 这种方式不会自动安装依赖。
- 如果 `webapp/static/index.html` 不存在，根路径 `/` 会尝试回退到模板渲染。
- 当前仓库并没有现成的 `webapp/templates/index.html`，所以后端单独运行时通常应保留已构建的 `webapp/static/`。

### 6.3 前端热更新开发

推荐联调方式：

1. 先启动后端固定在 `5000` 端口。
2. 再启动 Vite 开发服务器。

```powershell
Set-Location .\Game_screenshotReed_Autowork
..\.venv\Scripts\python .\webapp\app.py
```

另开一个终端：

```powershell
Set-Location .\Game_screenshotReed_Autowork\webapp\frontend
$env:VITE_API_BASE = "http://127.0.0.1:5000"
npm run dev
```

说明：

- `App.vue` 当前通过 `VITE_API_BASE` 拼接接口地址。
- `vite.config.js` 里仍保留了一部分历史代理规则，但没有覆盖所有新接口。
- 因此前端开发时，最稳妥的方式是显式设置 `VITE_API_BASE`，不要依赖现有代理配置。

### 6.4 前端构建

前端改动完成后，需要重新构建到 Flask 静态目录：

```powershell
Set-Location .\Game_screenshotReed_Autowork\webapp\frontend
npm run build
```

构建输出目录固定为：

`Game_screenshotReed_Autowork/webapp/static/`

这一步非常重要，因为：

- Flask 集成运行默认读取这里的产物。
- 打包脚本也会把这里的静态文件一并带入发布目录。

## 7. 配置文件说明

配置文件路径：

`Game_screenshotReed_Autowork/config.json`

后端默认配置定义在 `core/config.py`：

```json
{
  "adb_path": "",
  "default_instance": 1,
  "adb_port": 16384,
  "server_lang": "zh-CN",
  "emulator_type": "MuMu12",
  "emulator_configs": {},
  "screenshot_method": "PNG",
  "invitation_characters": ["", "", "", "", ""]
}
```

各字段含义如下：

- `adb_path`
  - ADB 可执行文件路径。
  - 支持相对路径，保存时会做标准化处理。
- `default_instance`
  - 当前仍保留在配置中，但主流程里更多依赖 `adb_port`。
- `adb_port`
  - ADB 连接端口。
  - MuMu12 常见值为 `16384`，雷电常见值为 `5555`。
- `server_lang`
  - 影响模板目录选择。
  - `zh-CN` 对应 `templates_zh-CN/`。
  - `zh-Hant` 对应 `templates_zh-Hant/`。
- `emulator_type`
  - `MuMu12`、`LDPlayer`、`Custom` 三选一。
- `emulator_configs`
  - 不同模拟器配置组。
  - 前端切换模拟器类型时会读写这里的 `path` 和 `port`。
- `screenshot_method`
  - 后端支持 `PNG`、`RAW`、`SCRCPY`。
  - 前端当前只暴露 `PNG` 和 `RAW`。
- `invitation_characters`
  - 邀约角色数组，长度固定为 5。

实现细节：

- `/config` 接口只接受 `DEFAULT_CONFIG` 里列出的键，未知字段会被忽略。
- `invitation_characters` 在设置页保存时会写回配置文件。
- 任务启动时若请求体中携带 `invitation_characters`，后端也会再次持久化。
- 切换 `server_lang` 后，需要重启任务对象或服务，模板语言才会稳定切换。

## 8. 任务类型与接口契约

### 8.1 `POST /task/start`

支持的任务类型：

- `start_game`
- `dailytasks`
- `tower_climbing`
- `combo`
- `daily_and_tower`
- `debug_sleep`
- `debug_loop`

其中：

- `combo` 表示“启动游戏 + 日常任务”。
- `daily_and_tower` 表示“日常任务 + 自动爬塔”。
- `debug_sleep`、`debug_loop` 当前主要用于本地暂停/恢复联调，前端未暴露。

常见请求体示例：

```json
{
  "type": "daily_and_tower",
  "daily_sub_tasks": [
    "interaction",
    "market_reward",
    "commission",
    "friend_manage",
    "gift",
    "invitation",
    "card_upgrade",
    "character_upgrade",
    "task_reward"
  ],
  "invitation_characters": [
    "gerui",
    "xiahua",
    "lalu_shengye",
    "wuyu",
    "feilencui"
  ],
  "add_friend_count": 10,
  "attribute_type": "light_earth",
  "climb_type": "standard",
  "max_runs": 0,
  "keep_tower_record": false
}
```

字段说明：

- `daily_sub_tasks`
  - 日常任务子项列表。
  - 当前有效值：
    - `interaction`
    - `market_reward`
    - `commission`
    - `friend_manage`
    - `gift`
    - `invitation`
    - `card_upgrade`
    - `character_upgrade`
    - `task_reward`
- `invitation_characters`
  - 最多 5 个角色 key。
- `add_friend_count`
  - 好友管理步骤中的加好友数量。
- `attribute_type`
  - `light_earth`
  - `water_wind`
  - `fire_dark`
- `climb_type`
  - `quick`
  - `standard`
- `max_runs`
  - `0` 表示不限次数，由周常完成条件控制结束。
- `keep_tower_record`
  - 爬塔结束后保留记录而不是走删除流程。

### 8.2 任务状态接口

- `GET /task/status`
  - 返回：
    - `state`
    - `task`
    - `running`
    - `canStop`
    - `canPause`
    - `canResume`
- `POST /task/pause`
- `POST /task/resume`
- `POST /task/stop`

### 8.3 预览与截图接口

- `GET /video_feed`
  - MJPEG 图像流。
- `GET /stream/start`
  - 开启预览并占用截图流消费者。
- `GET /stream/stop`
  - 停止预览并释放消费者。
- `GET /stream/status`
  - 可用于排查当前截图模式与底层状态。
  - 关键字段包括：
    - `running`
    - `method`
    - `effective_method`
    - `last_capture_backend`
    - `stream_alive`
    - `frame_age`
    - `consumers`
- `POST /api/test_latency`
  - 测试一次真实截图延迟。

### 8.4 配置接口

- `GET /config`
- `POST /config`
- `POST /config/test_adb`

`/config/test_adb` 的请求体格式：

```json
{
  "adb_path": "E:\\Program Files\\Netease\\MuMu Player 12\\shell\\adb.exe",
  "adb_port": 16384
}
```

### 8.5 日志与健康接口

- `GET /logs`
  - 支持 `?since=<idx>` 增量拉取。
- `GET /logs/export`
- `GET /health`

### 8.6 更新接口

- `GET /system/version`
- `GET /system/check_update`
- `POST /system/start_update`

### 8.7 兼容保留接口

当前后端还保留以下旧入口：

- `POST /start_game`
- `POST /start_dailytasks`

新功能建议优先走统一任务接口 `/task/start`。

## 9. 前后端协作点

### 9.1 前端主入口

前端主界面当前集中在：

`Game_screenshotReed_Autowork/webapp/frontend/src/App.vue`

这个文件同时负责：

- 标签页切换
- 任务参数收集
- 状态轮询
- 日志轮询
- 预览开启/暂停
- 设置页保存和连接测试
- 检查更新

### 9.2 新增功能时通常要改哪些地方

如果新增一个“日常任务子项”，通常至少要同步修改：

1. `core/daily_tasks.py`
2. `webapp/app.py` 中 `/task/start` 参数透传逻辑
3. `webapp/frontend/src/App.vue` 中勾选项和请求体构造
4. 如果依赖模板识别，还要补齐 `templates_zh-CN/` 和 `templates_zh-Hant/`

如果新增一个“配置项”，通常至少要同步修改：

1. `core/config.py` 的 `DEFAULT_CONFIG`
2. `webapp/app.py` 的 `/config` 使用逻辑
3. `webapp/frontend/src/App.vue` 设置页
4. 必要时更新打包后的默认 `config.json`

如果新增一个“截图模式或调试信息”，通常至少要同步关注：

1. `core/mumu_screenshot.py`
2. `webapp/app.py` 的 `/stream/status` 和 `/api/test_latency`
3. `webapp/frontend/src/App.vue` 的设置页和监控页

## 10. 打包、发布与更新

### 10.1 打包流程

`Game_screenshotReed_Autowork/package.ps1` 的实际步骤如下：

1. 使用 `..\.venv\Scripts\pip` 安装打包依赖。
2. 进入 `webapp/frontend` 执行 `npm ci` 和 `npm run build`。
3. 使用 PyInstaller 构建主程序。
4. 手动拷贝运行时资源到 `dist/StellasoraMaster/`。

脚本重点会额外复制：

- `core/`
- `run_app.py`
- `templates_zh-CN/`
- `templates_zh-Hant/`
- `config.json`
- `version.txt`
- `webapp/app.py`
- `webapp/__init__.py`
- `webapp/static/`

说明：

- `StellasoraMaster.spec` 中显式收集了 `scrcpy`、`av`、`adbutils`、`flask`、`cv2`、`numpy`、`ddddocr`。
- `spec` 里又排除了 `core`、`webapp`、`run_app`，因此打包脚本必须再手动复制这些运行时资源。

### 10.2 更新流程

更新链路分散在两个位置：

- `webapp/app.py`
  - 提供更新检查接口和启动更新接口。
- `update.py`
  - 负责真正下载并覆盖安装目录。

当前更新服务器地址为硬编码：

- `webapp/app.py` 中 `/system/check_update`
- `update.py` 中 `SERVER_URL`

如果后续更换更新服务器，请至少同步检查：

- 版本 JSON 地址
- 下载主机名
- 补丁包目录结构
- `version.txt` 的生成和覆盖逻辑

## 11. 维护建议与已知现状

### 11.1 关于前端产物

- `webapp/static/` 是实际运行依赖的前端构建产物，不只是临时目录。
- 修改 `webapp/frontend/src/` 后，如果没有重新执行 `npm run build`，一体化运行和打包产物都不会拿到最新前端代码。

### 11.2 关于模板资源

- `BaseTask` 初始化时会根据 `server_lang` 绑定模板目录。
- 若只维护简中模板、不维护繁中模板，切到 `zh-Hant` 时相关功能会直接失效。

### 11.3 关于截图模式

- 当前最推荐模式是 `RAW`。
- `PNG` 更稳但更慢。
- `SCRCPY` 在后端仍可用，但 UI 不再直接暴露，且不稳定时会自动降级。

### 11.4 关于测试现状

- 当前仓库没有成体系的自动化测试脚本或测试用例。
- `test/` 和 `tower_climber_test/` 目前更多是预留调试目录。
- 任何重要改动都建议至少手工验证：
  - 配置读写
  - ADB 连接测试
  - 截图延迟测试
  - 任务启停、暂停、恢复
  - 前端构建后是否能正常由 Flask 提供

### 11.5 关于 `run_app.py` 自动装包

- 这是开发便利能力，不应替代正式环境初始化。
- 团队协作时仍建议统一使用根目录 `requirements.txt` 初始化环境，再运行项目。

## 12. 建议的日常开发流程

推荐维护者采用下面这套流程：

1. 在仓库根目录准备并激活 `.venv`。
2. 安装根目录 Python 依赖。
3. 在 `webapp/frontend/` 安装 Node 依赖。
4. 后端调试时优先用 `webapp/app.py` 固定 `5000` 端口。
5. 前端调试时使用 `VITE_API_BASE=http://127.0.0.1:5000` 启动 Vite。
6. 前端改动完成后执行 `npm run build`。
7. 需要验证完整运行形态时，再用 `launcher.py` 或 `run_app.py` 做一体化验证。
8. 需要发布时执行 `package.ps1`。
